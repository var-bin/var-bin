Менеджеры терминалов. Screen

screen доступен на песочницах с настройками по умолчанию. Чтобы его запустить, достаточно выполнить команду screen.
Пример файла конфигураций: 
			# Select whether you want to see the copyright
			# notice during startup.
			# Default is ‘on’, as you probably noticed.
			startup_message off

Недостатки:
  • Не умеет создавать layout для каждой вкладки;
  • Не умеет сохранять layout, если оборвалась связь или еще что-то там;
  • До версии 4.1 не умел делать разбивку по вертикали (для меня это критично);
  • Когда вы создаете новый регион, то в нем отдельно нужно создавать новое окно (Ctrl+a c), выполнить переход в нужную директорию и т.д.;
  • Чтобы удалить регион тоже нужно помучатся Ctrl+a :remove.

Менеджеры терминалов. Tmux

“Это тот-же screen, только на стероидах”
Сборка tmux:
  1. wget --spider https://github.com/tmux/tmux/releases/download/2.2/tmux-2.2.tar.gz - проверим доступность файла по ссылке
  2. wget -P ~/ https://github.com/tmux/tmux/releases/download/2.2/tmux-2.2.tar.gz - если все норм, скачиваем архив себе в домашнюю директорию

Менеджеры терминалов. Tmux

Сборка tmux:
  1. tar -xzvf ~/tmux-2.2.tar.gz - нам нужно разархивировать содержимое архива, а сам архив можно удалить
  2. ./configure && make - нам нужно выполнить сборку, чтобы можно было обращаться по имени tmux

Менеджеры терминалов. Tmux

Преимущества tmux:
  1. tmux что называется “с коробки” настроен довольно качественно;
  2. не нужно настраивать положение панели с вкладками и как их подсвечивать, как это было с screen;
  3. умеет делить экран на так называемые регионы, как по вертикали, так и по горизонтали;
  4. умеет держать удаленное соединение *.
* нужна дополнительная магия для того, чтобы tmux сохранил доступы

Скрипты на bash. Основы. Перменные

Перменные:
  • имя - буквы, цифры, _.
  • значение - числа, строки (если есть пробелы, то в кавычках), отдельные символы.
Создание (перезапись) переменной:
path=~/Docs
Чтение переменной:
$path или ${path}

Скрипты на bash. Основы. Аргументы

Передача аргументов скрипту:
./script.sh arg1, arg2, arg3, …, argN
Обработка внутри скрипта:
$1 - первый аргумент
$2 - второй аргумент
$0 - имя скрипта
$# - количество аргументов

Быстрый дифф

Стандартный подход
git diff origin/master origin/%branch-name% > ~/%path_to_diff_dir%/diff-%branch-name%.diff
%branch-name% # имя ветки для которой нужно создать diff
В данном подходе есть как минимум 2 больших минуса:
  • уходит много времени на набор данных команд с клавиатуры;
  • в процессе набора большой % того, что мы допустим ошибки в командах и нужно будет все исправлять, что тоже занимает время

Быстрый дифф

Автоматизируем
  • gdd %branch-name% - финальный вид команды
  • написать скрипт
  • добавить команду gdd
  • Показать код

Добавляем миграции пачками

Каждый раз нужно было писать arm migration %migration_name%. Прошу обратить внимание - каждую миграцию нужно добавлять отдельно.
Решение, которое я себе представлял:
  • ввести команду в терминале и передать ей список миграций через пробел;
  • если количество миграций равно 1, значит добавляем миграцию и все;
  • если больше 1, значит запускаем цикл и в цикле добавляем все миграции;
  • дополнительные плюшки - добавить информацию о названии скрипта, количество миграций, которые были добавлены и названия самих миграций;

Up репозиториев в %path_to_repositories%

Иногда бывают проблемы с репозиториями, которые находятся в директории %path_to_repositories%
Проблема:
  • очередная миграция или какие-то обновления в этой директории, которые не описаны нигде с большой долей вероятности приведут вашу песочницу в неработающее состояние

Up репозиториев в %path_to_repositories%

Решение:
  • Чтобы исправить положение нужно перейти в эту директорию и проверить все репозитории на предмет того, чтобы они все были на live ветке и в состоянии Already up-to-date. Бывают даже проблемы, когда репозиторий находится на ветке live, но не в состоянии Already up-to-date. И тогда тоже нужно вручную сделать git pull origin live.
